📌 Getting Started with GLANCER
GLANCER extends Blockly by introducing a Glassbox debugging approach, Enabling synchronized debugging between the No-Code workspace and multiple programming languages.
  • Use the Blockly workspace to visually build programs.
  • The Glassbox Code View shows generated code in real-time (read only).
  • Select which programming languages to display and start developing.
  • Check out our proposed demos below.
🔴 Setting Breakpoints
Breakpoints can be set on any block or source-line-of-code.
GLANCER synchronizes the the breakpoints between the visual and textual layers
  • Block Breakpoint set: Right-click any block → "🔴 Add Breakpoint" to pause execution when block is reached. Similarly disable or remove it.
  • Code breakpoint set: Click left of a code line number to set a breakpoint, as popular with traditional code editors. Breakpoint clicks are cyclic: Enabled → Disabled → Removed
  • Breakpoints automatically sync between blocks and code lines.
  • Add or remove or disable breakpoints
🐞 Debugging
  • Use the Debugger Controls at the navigation bar to execute the program debugger.
  • Execution pauses at breakpoints, highlighting current location on both layers
  • Use the various step controls for precise execution control.
  • A variable and watches state table appears when debugging.
🔎 Highlighting Blocks & Code
Orient quickly between layers by highlighting any block or code line and view the corresponding elements highlight.
  • Right-click any block → Highlight, and view the corresponding SLOC highlight.
  • Mouse middle-click a code line number to highlight the corresponding block. Another click will remove the highlight.
📁 Snapshots & Execution Logs
  • Click the Snapshots menu at the top navbar.
  • Snapshots store program source and breakpoints.
  • Execution logs and Snapshots auto-create after each debugging session.
  • Snapshots can be saved manually at any time.
  • Choose a snapshot to preview and load it. The program will update according to the snapshot and saved breakpoints.
🚚 Exporting & IDE Integration
  • Export breakpoints for external IDE debugging (VS Code with BreakpointIO extention) at any of the included target programming languages.
  • Copy generated code for further development and debugging on the IDE.
🧪 Unit Testing
  • Use Unit Test Blocks to wrap statements and verify correctness.
  • Tests show pass/fail results visually.
  • Logs store test results for debugging analysis.
▶ Multi-Langague Remote Execution
Since the debugger executes JavaScript on the browser, we allow remote execution of program code in more programming langauges.
  • Use the corresponding navbar menu to remotley execute the program code in multiple programming languages and view results comparison.
  • Since different programming languages might behave differently, results may vary.

This demo aims to set the current traffic light color to the first color of the three.

Guide:
  1. Load the problem by clicking the "Try Debugging" button.
  2. Run the program and observe results.
  3. Execute the program in multiple programming languages by selecting the "Multi-Language Execution" option in the navigation bar and compare results.
  4. Experiment with the program parameters and view the logs by selecting the "Logs Menu" in the navigation bar and compare different runs.
Debugging Clues:
  • Clue #1: Pay attention to the code line that assigns a value to the variable current_light.
  • Clue #2: In Python, accessing a list with an index of -1 retrieves the last element of the list (reverse list access).
  • Clue #3: In Blockly, lists are 1-indexed, whereas in JavaScript, lists are 0-indexed.

Conclusions:
  1. By comparing Blockly with generated code, we can immediately notice differences in list indexing behavior across various programming languages.
  2. Executing the program in different programming languages demonstrates how they handle common cases of list out-of-bounds access.
  3. Through this exercise, you've encountered several programming issues and insights:
    • Out-of-bounds access
    • 0-indexing vs. 1-indexing
    • The abstraction gap between Blockly and its generated code

This demo showcases a proposed solution for finding the greatest common divisor (GCD) of two numbers, specifically 24 and 36.

Guide:
  1. Load the problem by clicking the "Try Debugging" button.
  2. Run the program without breakpoints and observe the infinite loop.
  3. Manually stop the program.
Debugging Clues:
  • Clue #1: Notice that in all programming languages, num1 and num2 are initialized as Not-a-Number (NaN), not 24 and 36 as defined in Blockly.
  • Clue #2: Set breakpoints within the loop and monitor the variable values in the table.
  • Clue #3: Examine the loop's termination condition. What happens when num2 becomes zero?
  • Clue #4: num1 should receive the original value of num2 after num2 is assigned the calculated remainder.

Conclusions:
  1. By viewing Blockly alongside code, we can immediately notice that the initial variable values are not set as intended.
  2. Debugging the program, which exhibits an infinite loop, allows us to observe the variable states and how the code control flow operates in sync with Blockly.
  3. Through this exercise, you've encountered interesting points such as:
    • Abstraction gap between Blockly and its generated code.
    • Infinite loops
    • Implicit type conversion
    • Incorrect loop termination conditions
    • Semantic errors
